## 一、能力边界清单

| 环节 | 具体能力 | 工具链 |

### ⚠️ 我需要 Elbow 协助（无法独立完成）

| 环节 | 为什么需要求助 | 建议协作方式 |

| **交互验证** | 拖拽、滑动、动画、转场效果，我只能看静态截图 | 我编译安装后 → Elbow 在模拟器操作 → 截图/语音反馈给我 |

## 二、目标工作流

```
┌─────────────┐
│  Elbow 提需求 │
└──────┬──────┘
       │ 需求描述（文字/截图/语音）
       ▼
┌─────────────────────┐
│ 鸡哥：理解需求 + 调研 │  ← Web 搜索 HarmonyOS 方案
└──────┬──────────────┘
       │ 如有疑问，立刻问 Elbow（不瞎猜）
       ▼
┌─────────────────┐
│ 鸡哥：编写代码    │  ← 修改 .ets 文件
└──────┬──────────┘
       ▼
┌─────────────────┐
│ 鸡哥：编译构建    │  ← rebuild.bat
└──────┬──────────┘
       │ 编译失败 → 自行修复，最多重试 3 次
       │ 3 次仍失败 → 通知 Elbow 求助
       ▼
┌─────────────────┐
│ 鸡哥：安装+运行   │  ← hdc install + aa start
└──────┬──────────┘
       ▼
┌─────────────────┐
│ 鸡哥：截图确认    │  ← 确认 UI 布局、文字正确
└──────┬──────────┘
       │
       ├─ 纯静态 UI → ✅ 截图 OK 则直接推送代码
       │
       └─ 有交互功能 → ⚠️ 需要 Elbow 验证
              │
              ▼
       ┌─────────────────────────┐
       │ 鸡哥通知 Elbow：         │
       │ "编译安装完成，请验证 XX  │
       │  交互效果"                │
       └──────┬──────────────────┘
              │ Elbow 操作 + 反馈
              ▼
       ┌─────────────────┐
       │ 鸡哥根据反馈迭代 │  ← 修改代码 → 重新编译安装
       └──────┬──────────┘
              │ 直到满意
              ▼
       ┌─────────────────┐
       │ 鸡哥：Git 推送   │
       └─────────────────┘
```

## 三、沟通协议

### 鸡哥主动汇报的节点
1. **开始工作**："收到，开始搞 XX"
2. **代码写完**："代码写好了，开始编译"
3. **编译结果**："编译成功/失败（附错误信息）"
4. **安装完成**："已安装到模拟器，截图如下"
5. **需要验证**："静态 UI OK，但 XX 交互需要你验证"
6. **完成推送**："代码已推送，over"

### Elbow 反馈格式建议
- ✅ "OK，没问题" → 我继续下一步
- ❌ "XX 不对" → 附截图/描述，我定位修改
- 🎯 "XX 效果要改" → 描述想要的效果，我调整

### 超时处理
- 鸡哥单任务沉默 > 5 分钟 → 自动汇报进度
- Elbow > 10 分钟未回复 → 继续能做的部分（如写下一模块代码），不傻等

## 四、重试与止损规则

| 场景 | 规则 |
|------|------|
| 编译错误 | 自行修复，最多 3 次 → 仍失败则求助 |
| 安装失败 | 先 uninstall 再 install，最多 2 次 → 仍失败则求助 |
| 截图失败 | 直接用已验证命令，不反复试 |
| 同一问题 | 尝试 3 种不同方案均失败 → STOP，求助 Elbow |
| 交互问题 | 不瞎猜效果，直接请 Elbow 验证 |
